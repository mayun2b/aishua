# AI刷题系统开发思路文档

## 1. 文档目的

本文档基于当前数据库设计 `aishua.sql`、数据库说明文档 `aishua.txt` 以及现有实现说明，整理一份后续开发的整体思路。

目标不是重复表结构，而是明确：

- 系统应该先做哪些模块
- 每个模块的职责边界是什么
- 各模块之间如何衔接
- 开发顺序如何安排，才能尽快形成可用闭环
- 后续如何从“能跑”逐步演进到“可维护、可扩展”

---

## 2. 系统定位

这个项目本质上是一个“以学科为主维度、以题目为核心资源、以练习和考试为主业务场景”的智能刷题系统。

从数据库设计看，系统包含 8 个核心业务域：

1. 用户与认证
2. 学科与学习入口
3. 教材目录与考点标签
4. 题库管理
5. 练习与错题闭环
6. 学习统计与掌握度分析
7. AI 生成题目
8. 考试与试卷

这 8 个域不是并列开发的，应当围绕“先形成完整主链路”来推进。

---

## 3. 业务主链路

从用户视角看，系统的最核心业务链应当是：

1. 注册/登录
2. 选择或加入学科
3. 查看题目
4. 开始练习
5. 提交答案
6. 更新错题、统计、掌握度
7. 查看个人学习结果
8. 针对薄弱点继续训练

从管理员视角看，核心业务链应当是：

1. 管理员登录
2. 维护学科
3. 维护教材目录
4. 维护考点标签
5. 维护题库
6. 维护试卷
7. 查看用户学习与题库运营情况

因此，开发顺序必须围绕这两条链路展开，而不是简单按数据库表逐个实现。

---

## 4. 推荐开发阶段

### 第一阶段：基础平台层

目标：让系统具备最基本的可运行能力。

应完成内容：

- 用户注册、登录、获取个人信息
- JWT 鉴权与登录态管理
- 管理员端基础框架
- 前后端统一返回结构
- 全局异常处理
- 数据库初始化脚本可稳定执行
- 基础配置外置化

这一阶段的产出标准：

- 普通用户能登录
- 管理员能登录
- 前后端已经建立统一接口风格
- 后续模块可以直接在这个基础上追加

说明：

这一层已经是所有模块的底座，后面不应该再频繁推翻。

---

### 第二阶段：学科中心

目标：建立全系统的主维度。

应完成内容：

- 管理员端学科管理
- 用户端学科列表展示
- 用户加入学科/我的学科
- 学科启停状态控制
- 学科排序与展示优先级

涉及表：

- `subject`
- `user_subject`

这一阶段的意义：

- 学科是题库、目录、考点、练习、统计、考试的上层入口
- 如果学科层没定下来，后续所有数据归属都会混乱

当前建议：

- 先完成管理员维护学科
- 再补用户“加入学科”和“我的学科”页面

---

### 第三阶段：知识结构层

目标：让题库具备清晰组织方式。

应完成内容：

- 教材目录树管理
- 考点标签管理
- 目录与学科的绑定
- 考点与学科的绑定

涉及表：

- `textbook_directory`
- `exam_tag`

建议做法：

- 目录管理优先做树形结构的增删改查
- 考点管理优先做平铺管理，后续再做层级扩展
- 后端设计时要保证“目录”和“考点”都可以被题目引用

这一阶段的意义：

- 题目不应只是一个孤立资源
- 后续错题筛选、知识点掌握度分析、AI 补题，都依赖这层结构

---

### 第四阶段：题库管理

目标：完成管理员核心生产工具。

应完成内容：

- 题目新增、编辑、删除、分页查询
- 题目按学科筛选
- 题目按目录筛选
- 题目按难度筛选
- 题目绑定多个考点
- 题目导入测试数据

涉及表：

- `question`
- `question_tag_relation`

建议接口方向：

- 管理员接口按 `/api/admin/questions` 设计
- 前台练习查询接口单独走 `/api/question` 或 `/api/practice/questions`

关键设计要点：

- 题目表是全系统最核心资源表
- 编辑题目时要同时维护题目主表和题目-考点关联表
- 题目删除建议优先做逻辑删除，不建议物理删除主数据

---

### 第五阶段：练习闭环

目标：形成普通用户可持续使用的主业务。

应完成内容：

- 开始练习
- 随机抽题/顺序练习
- 按学科练习
- 按目录练习
- 提交单题答案
- 批量提交答案
- 自动判分
- 自动记录练习流水
- 自动收录错题
- 自动更新统计数据

涉及表：

- `practice_session`
- `exercise_record`
- `wrong_question`
- `user_stats`
- `user_subject_stats`
- `daily_stats`
- `user_knowledge_mastery`

这一阶段是系统的真正业务核心。

开发建议：

1. 先做最小闭环：
   用户开始练习 -> 获取题目 -> 提交答案 -> 返回判分结果
2. 再做数据联动：
   练习记录、错题记录、统计记录同步更新
3. 最后做增强模式：
   错题重练、限时练习、AI 强化练习

关键难点：

- 一次提交答案会改多张表
- 必须使用事务保证一致性
- 判题逻辑要按题型拆开，不要写成一个大 if-else 方法

---

### 第六阶段：统计与学习分析

目标：让系统不只是“刷题”，而是“有反馈的刷题”。

应完成内容：

- 个人总览统计
- 学科维度统计
- 每日打卡统计
- 错题数量统计
- 知识点掌握度统计
- 学习趋势图

涉及表：

- `user_stats`
- `user_subject_stats`
- `daily_stats`
- `user_knowledge_mastery`

建议展示维度：

- 总做题数
- 正确率
- 最近学习日期
- 连续学习天数
- 各学科正确率
- 薄弱考点

这一阶段建议在前端形成一个“学习中心”或“个人仪表盘”。

掌握度分级建议（当前项目已落地）：

- 统计粒度：按“用户 + 考点（tag）”累计 `total_count`、`correct_count`、`wrong_count`、`correct_rate`。
- 分级目标：当题库规模很大时，避免“仅做几题就被判定为精通”。
- 小样本阶段：当某考点作答数 `< 20` 时，先按正确率分级，但最高只允许到 `2` 级。
- 稳定阶段：当某考点作答数 `>= 20` 时，按正确率直接分级。
- 正确率阈值（默认）：
- `>= 90%` -> `3` 级
- `>= 75%` -> `2` 级
- `>= 60%` -> `1` 级
- `< 60%` -> `0` 级

说明：

- 以上阈值应配置化，支持按学段或业务策略调整。
- 分级规则需与前端文案保持一致（薄弱、起步、熟悉、稳固等）。

第六阶段任务细化清单（结合当前项目实现）：

1. 统计聚合接口（P0）
- 目标：统一学习统计入口，避免前端多接口拼装。
- 接口：`GET /api/practice/stats?days=7|30|90`，默认 `days=30`。
- 返回结构：`overview`、`dailyTrends`、`subjectStats`、`knowledgeMasteries`、`weakPoints`、`recentSessions`。
- 异常口径：`days` 不在 `[7, 90]` 返回 `400`。
- 当前落地：`PracticeController#getPracticeStats` + `PracticeServiceImpl#getPracticeStats`。

2. 统计数据联动（P0）
- 目标：确保“交卷即更新统计”，不出现前端看不到最新结果。
- 联动范围：`user_stats`、`user_subject_stats`、`daily_stats`、`user_knowledge_mastery`、`wrong_question`。
- 触发点：`submitPractice` 提交成功后，同事务更新所有统计与错题沉淀。
- 当前落地：`PracticeServiceImpl#submitPractice` 内的 `updateUserStats / updateUserSubjectStats / updateDailyStats / updateKnowledgeMastery / upsertWrongQuestion`。

3. 仪表盘展示层（P0）
- 目标：把“统计数据”转换成“可行动反馈”。
- 页面建议：
- 总览卡片：累计答题、整体正确率、今日练习、连续练习、错题沉淀、练习场次。
- 趋势图：最近 N 天做题数 + 正确率。
- 学科表现：按学科展示题量、正确率、耗时。
- 薄弱考点：按掌握度等级 + 错题量排序。
- 最近练习：最近场次、答题进度、正确率、耗时。
- 当前落地：`aishua-web/src/modules/dashboard/views/UserDashboardView.vue`。

4. 掌握度分级规则落地（P0）
- 目标：避免小样本误判“精通”，让分级更稳定。
- 规则：小样本 `< 20` 最高 `2` 级；样本 `>= 20` 按阈值直接分级。
- 阈值：`>=90% => 3`，`>=75% => 2`，`>=60% => 1`，`<60% => 0`。
- 配置项：`application.yml` 的 `practice.mastery.*`。
- 当前落地：`PracticeServiceImpl` 掌握度计算逻辑 + `PracticeMasteryProperties`。

5. 质量与回归验证（P1）
- 用例 1：同一天多次练习，`daily_stats` 聚合正确（题量、正确数、耗时）。
- 用例 2：跨天练习，连续天数与最大连续天数计算正确。
- 用例 3：同考点从小样本到稳定样本，掌握度等级按规则跃迁。
- 用例 4：删除/停用学科或考点后，统计页仍可展示“已删除学科/考点”兜底文案。
- 用例 5：空数据用户可看到完整空态，不报错。

6. 阶段内可延期项（P2）
- 周报/月报导出。
- 学习目标达成率（如每日 20 题目标）。
- 趋势图按学科/题型切换。
- 基于薄弱考点自动推荐练习集合。

---

### 第七阶段：错题体系增强

目标：让错题本从“列表”升级为“复习工具”。

应完成内容：

- 错题本分页
- 按学科筛选错题
- 按目录筛选错题
- 标记已掌握
- 错题重练
- 错题变化趋势

涉及表：

- `wrong_question`

说明：

- 错题模块虽然依赖练习模块，但它是留存和复习的重要抓手
- 从产品价值上看，这个模块优先级很高

---

### 第八阶段：考试模块

目标：在练习之外补齐正式测评能力。

应完成内容：

- 试卷管理
- 试卷题目配置
- 发起考试
- 考试作答
- 提交考试
- 考试总成绩
- 考试明细

涉及表：

- `exam_paper`
- `exam_paper_question`
- `exam_record`
- `exam_record_question`

建议开发顺序：

1. 先做试卷模板管理
2. 再做考试记录主流程
3. 最后再做考试统计与回顾分析

关键点：

- `exam_paper_question` 是模板层
- `exam_record_question` 是结果层
- 两者不能混用

---

### 第九阶段：AI 能力接入

目标：让系统从题库平台升级为智能学习平台。

应完成内容：

- AI 生成题目
- 薄弱点补题
- AI 强化练习入口
- AI 出题原因记录
- AI 题与正式题分离管理

涉及表：

- `ai_generated_question`

建议策略：

- AI 题目必须与正式题库隔离
- AI 出题先作为增强能力，不应反向污染正式题库
- AI 生成结果要保留生成原因、标签、是否已练习等信息

## 5. 后端开发思路

### 5.1 模块划分建议

推荐后端继续按“业务域”组织：

- `modules/user`
- `modules/subject`
- `modules/directory`
- `modules/tag`
- `modules/question`
- `modules/practice`
- `modules/wrong`
- `modules/stats`
- `modules/exam`
- `modules/ai`

每个模块建议统一包含：

- `controller`
- `service`
- `service/impl`
- `mapper`
- `entity`
- `entity/dto`
- `entity/vo`

### 5.2 基础层职责

基础层建议只保留公共能力：

- `common/result`
- `common/exception`
- `common/interceptor`
- `common/security`
- `common/context`
- `config`

不要把业务逻辑塞进 `common`。

### 5.3 开发原则

- 一个接口只处理一个明确业务动作
- 一个 service 方法尽量对应一个核心用例
- 多表修改的业务必须加事务
- 查询 VO 和写入 DTO 分离
- 题型判分逻辑单独封装，不直接散落在 controller

## 7. 接口设计建议

### 7.1 接口分层

建议接口按角色和用途区分：

- 用户侧接口：`/api/...`
- 管理侧接口：`/api/admin/...`

例如：

- `/api/user/login`
- `/api/user/profile`
- `/api/admin/subjects`
- `/api/admin/questions`
- `/api/practice/start`

### 7.2 命名原则

- 列表查询：`GET`
- 详情查询：`GET /{id}`
- 新增：`POST`
- 更新：`PUT /{id}`
- 状态切换：`PUT /{id}/status` 或 `/enabled`
- 删除：`DELETE /{id}`

## 8. 数据一致性重点

系统中最需要关注一致性的地方有 3 个：

### 8.1 提交练习答案

一次答题提交可能同时影响：

- `exercise_record`
- `wrong_question`
- `user_stats`
- `user_subject_stats`
- `daily_stats`
- `user_knowledge_mastery`
- `question` 中统计字段

因此必须使用事务。

### 8.2 删除题目

题目被删除后会影响：

- 题目-考点关系
- 错题引用
- 练习记录引用
- 考试题目引用

建议：

- 题目优先逻辑删除
- 后续再做清理策略

### 8.3 学科删除

学科关联范围很广：

- 目录
- 考点
- 题目
- 用户学科关系
- 统计
- 试卷

建议：

- 学科默认只做逻辑删除或禁用
- 不建议轻易物理删除

1. 学科管理完善
2. 用户加入学科/我的学科
3. 教材目录管理
4. 考点标签管理
5. 题库管理
6. 练习主流程
7. 错题本
8. 统计中心
9. 试卷与考试

说明：

- 只有题库结构稳定后，练习和考试模块才不会反复返工

## 10. 每个阶段的验收标准

### 阶段一验收

- 用户可注册登录
- 管理员可登录
- 登录态可维持

### 阶段二验收

- 管理员可维护学科
- 用户能看到学科并加入学科

### 阶段三验收

- 管理员可维护目录树和考点
- 目录和考点可以绑定到学科

### 阶段四验收

- 管理员可以维护题库
- 题目可绑定目录和考点

### 阶段五验收

- 用户可以开始练习并提交答案
- 系统自动更新错题和统计

### 阶段六验收

- 用户可通过 `GET /api/practice/stats` 获取完整统计对象，支持 `days=7/30/90`。
- 用户可在学习中心看到总览、趋势、学科表现、薄弱考点、最近练习五类数据。
- 提交练习后刷新统计页，关键指标（总题数、正确率、错题数、打卡数据）即时可见。
- 连续学习天数、最大连续天数在跨天场景下计算正确。
- 掌握度分级满足“小样本保护 + 阈值分级”规则，且阈值可配置。
- 空数据、新用户、学科/考点被删除等场景均有兜底显示，不影响页面可用性。

### 阶段七验收

- 用户可以使用错题本进行复习

### 阶段八验收

- 系统支持正式考试流程

### 阶段九验收

- 系统支持 AI 补题和个性化强化

## 11. 风险与注意事项

### 11.1 文档与 SQL 同步

当前已有设计文档存在编码问题和部分历史内容混杂的问题。

建议后续以 `aishua.sql` 为主，以本文档为开发路线说明，避免多个旧文档互相冲突。

### 11.2 先做闭环，不要同时铺太多模块

不要一开始就同时做题库、练习、考试、AI。

正确做法是：

- 先做最短业务闭环
- 再在闭环基础上补分析和增强能力

### 11.3 测试数据要尽早准备

题库类系统如果没有基础测试数据，前后端都会很难联调。

建议尽快准备：

- 2 到 3 个学科
- 每个学科 2 到 3 个目录
- 每个学科 5 到 10 个考点
- 每个学科 20 到 50 道测试题

---

## 12. 总结

这个项目的正确推进方式，不是“按表开发”，而是“按业务闭环开发”。

最优路径应当是：

1. 先把认证和管理底座打稳
2. 以学科为中心建立题库结构
3. 再完成练习闭环
4. 最后叠加统计、考试和 AI 能力

如果按这个顺序推进，系统会从：

- 能登录
- 能管理学科
- 能管理题库
- 能练习
- 能形成学习反馈
- 能做考试
- 能做 AI 强化

逐步成长为一个结构清晰、业务闭环完整的智能刷题平台。

## 13. RAG 驱动 AI 出题的专业方案（新增）

本章用于给“AI 强化练习”提供可直接落地的技术路线。  
目标不是单纯“让模型会出题”，而是建立一个“可诊断、可解释、可评估、可持续优化”的出题系统。

### 13.1 设计目标

建议将 AI 出题能力拆成 4 个目标层次：

1. 能出题：可以稳定生成结构正确、答案可判定的题目
2. 出对题：题目匹配用户当前薄弱点和目标难度
3. 出好题：题目质量可控，不重复、不跑题、可解释
4. 出有效题：上线后能显著改善正确率、错题回落率和学习留存

最终评估标准应是“学习效果提升”，不是“模型文本是否流畅”。

### 13.2 建议重点出题方向

结合当前题库与学习数据，优先做以下 6 类 AI 题：

1. 薄弱点修复题  
针对低掌握度考点（`mastery_level` 低、`correct_rate` 低）输出基础题与关键概念题。

2. 错因对抗题  
针对高频错题，生成同考点不同问法的变式题，避免“背答案”而不理解。

3. 难度爬坡题  
同一考点按简单 -> 中等 -> 困难递进，建立可持续进步路径。

4. 跨点综合题  
将两个或多个薄弱考点组合成综合题，训练知识迁移能力。

5. 考前冲刺题  
基于近期错题、历史高频考点和目标考试范围，自动组织冲刺题簇。

6. 遗忘复现题  
针对“曾掌握但长时间未练”的考点进行低频复现，降低遗忘曲线影响。

### 13.3 推荐技术架构（Hybrid RAG）

建议采用“结构化检索 + 向量检索 + 受控生成 + 质量闸门”的混合方案：

1. 数据层  
数据来源：`question`、`question_tag_relation`、`exam_tag`、`wrong_question`、`user_knowledge_mastery`、`exercise_record`、`daily_stats`。

2. 检索层  
第一阶段：结构化检索（学科、目录、考点、难度、题型、时间窗）。  
第二阶段：向量检索（题干、解析、知识点语义相似度）。

3. 重排层  
结合以下信号进行排序：  
- 用户薄弱度权重  
- 题目区分度（历史正确率、做题次数）  
- 新颖度（与已练题重复度惩罚）  
- 难度匹配度

4. 生成层  
当检索候选不足或需变式时调用大模型生成。  
模型输出强制为结构化 JSON，不接受自由文本结果。

5. 质检层  
自动校验：题型合法性、答案可判定、解析一致性、重复率阈值、敏感内容安全。

6. 投放层  
通过质检的 AI 题进入 `ai_generated_question`，并保留 `generate_reason`、标签、难度、是否练习。

### 13.4 核心实现原则

1. 检索优先，生成兜底  
优先命中高质量题库与变式模板，生成能力用于补足和强化，不作为唯一来源。

2. 题库隔离，防止污染  
AI 题与正式题库分层管理；正式题库不直接被 AI 输出反向覆盖。

3. 全链路可解释  
每道 AI 题必须记录“为什么生成给这个用户”，用于学习反馈和运营复盘。

4. 反馈闭环驱动优化  
按周分析 AI 题练习效果，动态调整出题策略参数，不只调模型 prompt。

### 13.5 建议补充的数据与索引（可扩展）

在保留当前主表的基础上，建议新增或扩展以下结构：

1. 题目向量索引表（或外部向量库）  
存储 `question` 和高质量 `ai_generated_question` 的 embedding 与元数据。

2. AI 出题策略快照表  
记录每次出题时的策略参数、检索条件、重排分数、prompt 版本。

3. AI 题质检结果表  
记录每道题的自动质检得分、失败原因、人工复核状态。

4. AI 题效果追踪表  
记录用户在 AI 题上的正确率提升、二次错误率、耗时变化等。

### 13.6 出题策略 DSL（建议）

建议将“出题逻辑”配置化，不要硬编码在 service 内。示例：

```json
{
  "scene": "weakness_repair",
  "subjectId": 3,
  "targetTags": [101, 205],
  "questionTypes": [1, 2, 3],
  "difficultyPlan": {"easy": 0.5, "medium": 0.4, "hard": 0.1},
  "count": 12,
  "noveltyThreshold": 0.82,
  "useGenerationWhenRecallLessThan": 8
}
```

这样可以在不改代码的情况下快速调策略。

### 13.7 推荐出题流程（端到端）

1. 输入用户与场景（薄弱修复/错题重练/冲刺等）
2. 读取用户画像（掌握度、错题、最近行为、目标难度）
3. 执行结构化检索召回候选题
4. 执行向量检索补充语义近邻题
5. 合并候选并重排
6. 候选不足时触发模型生成变式题
7. 执行自动质检和重复度过滤
8. 入库 `ai_generated_question` 并记录 `generate_reason`
9. 下发给练习会话（AI 强化模式）
10. 采集作答结果回写策略评估

### 13.8 评估体系（必须建设）

建议至少建立以下 4 组指标：

1. 质量指标  
- 结构合法率  
- 答案可判定率  
- 题目重复率  
- 人工抽检通过率

2. 学习效果指标  
- AI 题后同考点正确率提升  
- 二次错误率下降  
- 单题平均耗时变化  
- 7 日留存/复练率

3. 策略指标  
- 各场景命中率  
- 检索命中率与生成触发率  
- 不同难度配比下的收益差异

4. 成本指标  
- 单题生成成本  
- 平均响应时延  
- 向量检索与重排耗时

### 13.9 风险与控制建议

1. 幻觉风险  
控制：严格 JSON 输出 + 自动判题校验 + 不合格即丢弃。

2. 重复与同质化  
控制：向量去重 + 历史做题去重 + 变式模板多样化。

3. 难度漂移  
控制：上线前做难度标定，按真实作答反馈动态回归难度分层。

4. 策略失效  
控制：每周做 A/B 策略对比，保留可回滚的策略版本。

### 13.10 推荐推进顺序（AI 专项）

1. Phase A：可用版  
结构化检索出题 + 基础生成 + 入库 + 简单质检。

2. Phase B：增强版  
向量检索 + 重排 + 去重 + 质量评估看板。

3. Phase C：优化版  
策略 DSL + 自动调参 + 多场景出题引擎 + A/B 实验框架。

---

补充说明：  
AI 出题系统的核心竞争力不在“调用了哪个模型”，而在“是否形成了可持续迭代的教学反馈闭环”。

## 14. 错题 AI 分析与追问功能（新增）

本章用于在“错题记录”场景中落地 AI 诊断与多轮追问能力。  
目标是把错题本从“错题列表”升级为“可诊断、可提问、可复盘、可评估”的学习工具。

### 14.1 设计目标

建议将该能力分成 4 个可验收目标：

1. 能诊断  
用户点击错题后，系统可给出“错题原因 + 解题思路 + 避错建议”。

2. 能追问  
用户可以围绕同一错题持续提问，AI 能结合上下文连续回答。

3. 能沉淀  
每次分析和对话都可落库，支持学习复盘、成本统计、质量评估。

4. 能迭代  
基于使用数据持续优化 Prompt、上下文拼装策略和答案结构。

### 14.2 与现有系统的衔接原则

1. 主链路不破坏  
不改动现有练习提交主事务：`exercise_record`、`wrong_question`、`user_stats`、`daily_stats` 等保持稳定。

2. 能力解耦  
AI 能力采用“新增表 + 新接口”，不与答题判分事务强耦合。

3. 事务外调用  
模型调用放在事务外执行，避免模型超时影响练习主流程。

4. 结构化输出  
AI 输出必须强制为结构化 JSON，避免不可控自由文本污染业务数据。

### 14.3 核心用户流程（错题分析场景）

1. 用户进入错题记录列表
2. 点击“AI 分析”
3. 后端读取错题上下文（题目、标准答案、官方解析、错题标签、最近错答记录、掌握度）
4. 调用模型并返回结构化诊断结果
5. 诊断结果落库（分析快照）
6. 用户继续追问“某一步为什么这样做”
7. 系统进入错题对话会话，保存每轮消息
8. 用户退出后仍可回看历史分析与追问记录

### 14.4 需要新增的数据表（按已确认命名）

建议新增以下 3 张表：

1. `wrong_question_ai_analysis`  
用途：存储某道错题的一次 AI 首轮分析结果（含上下文快照、结构化结果、模型耗时与 token）。

2. `wrong_question_ai_chat_session`  
用途：存储“围绕某道错题”的一次会话头信息（会话状态、轮次、总 token、最后活跃时间）。

3. `wrong_question_ai_chat_message`  
用途：存储会话中的每一轮消息（user/assistant/system、内容、耗时、token、错误信息）。

### 14.5 字段设计建议（核心字段）

#### 14.5.1 `wrong_question_ai_analysis`

建议至少包含：

- `id`
- `analysis_code`（唯一编码）
- `user_id`
- `wrong_question_id`
- `question_id`
- `subject_id`
- `directory_id`
- `analysis_type`（首轮分析 / 重分析）
- `context_snapshot`（JSON，分析输入快照）
- `result_json`（JSON，结构化输出）
- `summary`（文本摘要）
- `model_provider` / `model_name` / `prompt_version`
- `prompt_tokens` / `completion_tokens` / `total_tokens`
- `latency_ms`
- `status`（成功/失败）
- `error_message`
- `create_time` / `update_time` / `deleted`

#### 14.5.2 `wrong_question_ai_chat_session`

建议至少包含：

- `id`
- `session_code`（唯一编码）
- `user_id`
- `wrong_question_id`
- `analysis_id`（可为空，关联首轮分析）
- `question_id`
- `subject_id`
- `status`（进行中/已结束）
- `round_count`
- `last_message_at`
- `total_prompt_tokens` / `total_completion_tokens` / `total_tokens`
- `create_time` / `update_time` / `deleted`

#### 14.5.3 `wrong_question_ai_chat_message`

建议至少包含：

- `id`
- `session_id`
- `seq_no`（会话内序号）
- `role`（system/user/assistant）
- `message_type`（text/json）
- `content_text`
- `content_json`
- `model_provider` / `model_name`
- `prompt_tokens` / `completion_tokens` / `total_tokens`
- `latency_ms`
- `status`
- `error_message`
- `create_time` / `update_time` / `deleted`

### 14.6 关联关系建议

1. `wrong_question_ai_analysis.wrong_question_id -> wrong_question.id`
2. `wrong_question_ai_chat_session.analysis_id -> wrong_question_ai_analysis.id`
3. `wrong_question_ai_chat_message.session_id -> wrong_question_ai_chat_session.id`

同时建议保留 `question_id`、`subject_id` 的冗余快照字段，降低后续统计联表成本。

### 14.7 索引与约束建议

1. 分析表  
- `uk_analysis_code`
- `idx_user_wrong_time (user_id, wrong_question_id, create_time)`
- `idx_question_time (question_id, create_time)`

2. 会话表  
- `uk_session_code`
- `idx_user_last_message (user_id, last_message_at)`
- `idx_wrong_last_message (wrong_question_id, last_message_at)`

3. 消息表  
- `uk_session_seq (session_id, seq_no)`
- `idx_session_time (session_id, create_time)`

### 14.8 接口设计建议（用户侧）

建议新增以下接口：

1. `POST /api/practice/wrong-questions/{wrongQuestionId}/ai-analysis`  
触发首轮 AI 分析，返回结构化分析结果。

2. `GET /api/practice/wrong-questions/{wrongQuestionId}/ai-analysis/latest`  
读取该错题最近一次分析结果。

3. `POST /api/practice/wrong-questions/{wrongQuestionId}/ai-chat/sessions`  
创建追问会话。

4. `GET /api/practice/wrong-questions/{wrongQuestionId}/ai-chat/sessions/{sessionId}/messages`  
读取会话消息历史。

5. `POST /api/practice/wrong-questions/{wrongQuestionId}/ai-chat/sessions/{sessionId}/messages`  
发送一轮追问并返回 AI 回复。

6. `PUT /api/practice/wrong-questions/{wrongQuestionId}/ai-chat/sessions/{sessionId}/close`  
结束会话。

### 14.9 AI 输入与输出规范

#### 输入上下文建议

至少包含：

1. 题目基础信息：题干、题型、选项、标准答案、官方解析
2. 错题信息：错题次数、最近错题时间、错题标签
3. 用户作答信息：最近 1~3 次同题错答记录（答案、耗时、时间）
4. 学习画像信息：关联考点掌握度、学科维度正确率
5. 当前提问：用户本轮问题 + 有限历史消息

#### 输出结构建议

强制模型返回 JSON，建议字段：

1. `error_reasons`：错因分类列表
2. `reason_evidence`：错因判断依据
3. `solution_steps`：解题步骤
4. `knowledge_points`：涉及考点
5. `avoidance_tips`：避免再错建议
6. `follow_up_questions`：建议用户继续追问的问题

### 14.10 前端改造建议（错题记录页）

1. 错题列表新增“AI 分析”按钮
2. 点击后弹出分析面板（首屏展示错因 + 解题步骤）
3. 面板下方提供追问输入框和会话区
4. 会话区展示 user/assistant 多轮消息
5. 提供“重新分析”“结束会话”“查看历史分析”入口
6. 增加提示语：AI 内容用于学习参考，最终以教材与老师讲解为准

### 14.11 分阶段开发建议（与现有开发顺序兼容）

1. Phase A：MVP 可用版  
完成三张表、首轮分析接口、错题页“AI 分析”展示。

2. Phase B：追问版  
完成会话接口、消息持久化、多轮追问、历史会话回看。

3. Phase C：运营与评估版  
补齐调用成功率、耗时、token 成本看板；增加 Prompt 版本对比与策略优化。

### 14.12 风险与控制

1. 模型幻觉  
控制：结构化输出 + 字段校验 + 失败重试 + 不合格结果标记失败态。

2. 成本失控  
控制：限制上下文长度、限制单会话轮次、记录 token 并做阈值告警。

3. 响应超时  
控制：超时降级提示、异步重试、会话消息失败态可见。

4. 越权访问  
控制：所有接口强校验 `wrong_question.user_id == 当前用户`。

5. 主链路受影响  
控制：AI 接口独立于练习提交事务，失败不影响错题主数据。

### 14.13 验收标准（本功能专项）

1. 功能验收  
用户可完成“错题 AI 分析 -> 连续追问 -> 历史回看”完整闭环。

2. 数据验收  
三张表数据完整可追溯，消息顺序正确，错误态可识别。

3. 性能验收  
在可接受时延内返回分析与追问结果，并有清晰超时提示。

4. 安全验收  
用户不能访问他人错题分析与会话数据。

5. 运营验收  
可统计调用次数、成功率、平均耗时、平均 token、用户追问轮次分布。

### 14.14 上线前必须修正的两个一致性问题

1. 练习模式枚举不一致  
`practice_mode` 在数据库注释、常量定义、DTO 校验范围需统一。

2. 模型密钥管理  
AI 平台 API Key 不应明文写入配置文件，应迁移到环境变量或密钥管理系统。
